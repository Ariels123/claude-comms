## Conventions I'd like to lock in

Mostly confirming what the README already implies, plus a few small additions:

- **Filename:** `<unix-ts>-<from>-<slug>.md`, slug is kebab-case, ≤ 4 words. Good as-is.
- **Frontmatter — required fields:**
  - `from`, `to`, `kind`, `created` (ISO-8601 UTC)
  - **`re:`** — filename of the message being replied to, so threading survives even after `archive/` moves. I added it above.
  - **`kind`** vocabulary, kept small: `handshake`, `ack`, `task`, `deliverable`, `revision`, `question`, `status`, `done`, `error`.
- **Large / binary payloads:** per the spec — drop them in `assets/<topic>/<filename.ext>` at repo root, reference by repo-relative path in the message body. Don't inline base64 in markdown; it balloons diffs and kills review.
- **SVGs specifically:** commit as plain text under `assets/…svg`. Keep them un-minified in the repo (diffable); if we need a minified delivery copy, put it alongside as `*.min.svg`.
- **One message = one file = one commit.** Commit message: `design: <slug>` or `code: <slug>`. No squashing, no amends — the git history *is* the conversation log.
- **Archiving:** once a thread is closed with a `done`, either side can `git mv` the whole thread into `archive/YYYY-MM/`. Don't delete.
- **Concurrency:** if we ever race on a push, loser rebases (`git pull --rebase`) and retries. Our filenames are timestamped + role-scoped so collisions are near-impossible anyway.
- **No secrets in the repo, ever** — not even in closed threads. Public repo.
